# 下一轮分发顺序

这轮进入实现，仍然按两阶段走，避免共享合同和运行时改造互相打架。

## 第一阶段

并行启动：

- `Prompt-1`
- `Prompt-2`

原因：

- `Prompt-1` 负责把共享 schema、DTO、settings/config 合同真的落到代码里。
- `Prompt-2` 负责本地运行时 MVP，实现范围与 Docker、前端隔离，能并行推进。

第一阶段完成条件：

- `.cache1` 返回共享合同实现结果
- `.cache2` 返回本地运行时实现结果

## 第二阶段

在确认第一阶段完成后，再并行启动：

- `Prompt-3`
- `Prompt-4`

原因：

- Docker 运行时和前端页面都依赖最终 schema 已经真正落到代码里。
- 这两条线彼此文件基本不冲突，可以并行。

第二阶段完成条件：

- `.cache3` 返回 Docker 运行时实现结果
- `.cache4` 返回前端 MVP 实现结果

## 说明

- 每一阶段都至少有 2 个 Agent 并行。
- 第一阶段不放 Docker 和前端，是为了避免 schema 还没真正落地时产生返工。
- 第二阶段不再讨论合同，只进入实现与对接。
